1. Arquitecto de Software

A) Errores y Bugs

CRÍTICO Archivo monolítico sin separación de responsabilidades
Todo el sistema (4.260 líneas de HTML + CSS + JS) vive en un único archivo. Esto hace imposible testear, cachear o desplegar parcialmente. Cualquier cambio requiere recargar todo el sistema.
CRÍTICO Estado global compartido sin protección
Todas las estructuras de datos (ausenciasGV, sustituciones, modificacionesHorario, EMPLEADOS_GV, etc.) son variables globales mutables. Cualquier extensión de navegador o inyección de consola puede corromper el estado completo.
CRÍTICO Sin capa de persistencia local
Si la conexión a Sheets falla durante una sesión, todos los cambios hechos (ausencias, sustituciones) se pierden al recargar. No hay localStorage/IndexedDB como fallback.
IMPORTANTE Race conditions en sincronización
Las funciones sync*() hacen fetch sin encolar ni esperar respuesta. Si el usuario hace dos cambios rápidos, las escrituras pueden solaparse y la segunda machacar la primera (el patrón es "guardar hoja completa", no append incremental).
IMPORTANTE No hay versionado de datos
Si dos personas abren la app simultáneamente y ambas modifican ausencias, el último en guardar sobrescribe todo sin merge ni detección de conflictos.
IMPORTANTE API endpoint hardcodeado en el cliente
La URL del Apps Script está expuesta en el HTML (línea 3364). Cualquiera con acceso al archivo puede leer/escribir datos.

B) Mejoras Técnicas

CRÍTICO Separar en módulos
Como mínimo: data.js (modelo/estado), rotaciones.js (motor de cálculo), ui.js (renderizado), sync.js (Google Sheets), sustituciones.js (motor automático). Idealmente con un bundler.
IMPORTANTE Patrón de estado centralizado
Reemplazar las variables globales por un store con getters/setters y eventos de cambio. Permitiría undo/redo, auditoría de cambios, y persistencia automática.
IMPORTANTE Capa de abstracción para backend
Interfaz DataSource que permita cambiar Google Sheets por Firebase, Supabase, o cualquier otro backend sin tocar lógica de negocio.
DESEABLE Migrar a ES6+ con transpilación
El código usa var, for loops con indexOf, y evita let/const, arrow functions, Array.map/filter, template literals, etc. Modernizar mejoraría legibilidad y mantenibilidad.

C) Mejoras Funcionales

IMPORTANTE Multi-usuario con roles
No hay concepto de "quién está haciendo el cambio". Si Eva y Nacho usan la app al mismo tiempo, no hay trazabilidad.
IMPORTANTE Modo offline
Service Worker + cache local para que funcione sin conexión y sincronice al recuperar red.
DESEABLE Histórico de cambios (audit trail)
Registrar quién hizo cada cambio, cuándo, y poder revertir.

D) Innovaciones

IMPORTANTE Arquitectura PWA
Convertir en Progressive Web App instalable, con push notifications para alertas de cobertura.
DESEABLE Microservicios ligeros
Un pequeño backend (Cloudflare Workers, Vercel Functions) que actúe como proxy autenticado a Sheets y gestione conflictos de concurrencia.
DESEABLE Componentes web reutilizables
Preparar la UI con Web Components o un framework ligero (Svelte, Alpine.js) para poder escalar a más tiendas sin duplicar código.

2. Ingeniero de Software Senior

A) Errores y Bugs

CRÍTICO contarDiasAusencia() cuenta días naturales, no laborables
La función (líneas 2513-2522) cuenta TODOS los días entre "desde" y "hasta", incluyendo sábados y domingos. Un empleado que pide vacaciones de lunes a viernes gasta 5 días, pero si es de lunes a domingo le cuentan 7. Esto infla artificialmente los días usados.
CRÍTICO cancelarAusencia() elimina sustituciones de otros empleados
El filtro (líneas 1864-1871) s.fecha >= a.desde && s.fecha <= a.hasta && s.tienda === tiendaActual borra TODAS las sustituciones en ese rango de fechas para esa tienda, no solo las del empleado ausente. Si hay dos ausencias solapadas en fecha, cancelar una borra las sustituciones de la otra.
CRÍTICO Cambio de año rompe semana ISO → semana A/B
getSemanaAB() (líneas 890-893) usa numSem % 2 para determinar A/B. En la transición diciembre→enero, la semana ISO puede ser 52 o 53, y la semana 1 del nuevo año puede empezar en diciembre. Esto puede causar que dos días consecutivos de la misma semana laboral tengan semana A/B diferente.
IMPORTANTE editarHorarioSustituto() usa prompt() con formato decimal
(Líneas 2406-2413) El usuario debe introducir horas como "10.5" para las 10:30. No hay validación del input. Un NaN corrompe la sustitución silenciosamente.
IMPORTANTE eliminarFalta() tiene índice frágil
(Líneas 2699-2723) El mapeo de índice visual a índice real depende de que el orden de faltas no cambie entre el renderizado y el click. Si otra pestaña guarda faltas entre medio, se elimina la falta incorrecta.
IMPORTANTE Rotación Isabel L-V referencia semana ISO 15 hardcodeada
(Línea 667) ((numSem - 15 + 2) % 4 + 4) % 4 funciona para 2026, pero si el negocio sigue en 2027, la referencia a semana 15 como "semana 3 de rotación" será incorrecta.
IMPORTANTE Detección de solapamientos incompleta en FdS
Si un empleado tiene turno sábado tarde y domingo mañana en diferentes tiendas y son consecutivos sin solapamiento horario, el sistema no detecta que el empleado no tiene descanso entre turnos.
IMPORTANTE Lógica ABEL/ANDREA condicional enterrada en fallback
(Línea 690) if (dow === 1) ABEL; else ANDREA; — Esta regla de negocio está oculta en el fallback hardcoded. Si Sheets falla, esta regla puede sorprender.
DESEABLE Cache de horas semanales potencialmente obsoleto
(Líneas 1342-1358) Se invalida solo por mes. Si el usuario añade una sustitución sin cambiar de mes, el caché puede devolver horas incorrectas.

B) Mejoras Técnicas

CRÍTICO Código duplicado masivo
La lógica de "obtener horarios para una fecha" se repite en al menos 15 funciones con variaciones. El patrón if(tiendaActual==='granvia') HORARIOS_GV[semAB]... else getHorariosIS(fecha) aparece ~20 veces.
CRÍTICO Generación de HTML con concatenación de strings sin escapar
Todo el renderizado construye HTML con += y datos sin escapar. Si un nombre de empleado contiene comillas o caracteres especiales (ej: "O'Brien"), el HTML y los onclick se rompen. Potencial vector XSS.
IMPORTANTE Funciones comprimidas ilegiblemente
(Líneas 1162-1168) Funciones de varias operaciones escritas en una sola línea. Dificulta depuración y lectura.
IMPORTANTE Búsquedas lineales O(n) en arrays
estaAusente(), getSustituto(), estaSustituyendo(), etc. iteran arrays completos por cada empleado y día. En un mes: ~1800 llamadas, cada una iterando los arrays. Funciona ahora pero escala mal.
IMPORTANTE Motor de sustituciones con complejidad N²
calcularSustituciones() itera todos los turnos sin cubrir y para cada uno evalúa todos los candidatos, iterando sustituciones simuladas. Con muchas ausencias simultáneas, podría ser lento.
DESEABLE Dos funciones duplicadas de número de semana
getWeekNumber() (línea 888) y getNumSemana() (línea 889) hacen esencialmente lo mismo con implementaciones ligeramente diferentes.

C) Mejoras Funcionales

IMPORTANTE Validar datos cargados de Sheets
No hay validación cuando se cargan datos de Google Sheets. Si una celda tiene un valor incorrecto (texto donde se espera número), el sistema falla silenciosamente.
IMPORTANTE No hay deshacer (Ctrl+Z)
Cualquier acción (asignar sustituto, cancelar ausencia) es irreversible excepto rehaciendo el paso inverso manualmente.
DESEABLE Tests unitarios para el motor de rotaciones
Las funciones getTurnoRotacion(), getTurnoABC(), getFdsGV(), getFdsIS() son lógica pura sin side effects. Ideales para test unitarios.

D) Innovaciones

IMPORTANTE Motor de sustituciones en Web Worker
El cálculo automático podría ejecutarse en background sin bloquear la UI.
DESEABLE Linting y formateo automático
Configurar ESLint + Prettier para mantener calidad de código consistente.
DESEABLE TypeScript o JSDoc
Tipar las estructuras de datos (empleado, ausencia, sustitución, rotación) para detectar errores en compile time.

3. Product Owner / Analista Funcional

A) Errores y Bugs

CRÍTICO El conteo de vacaciones incluye fines de semana
Un empleado que pide "lunes a lunes" (8 días naturales), consume 8 días en vez de 6 laborables. Esto genera conflictos con RRHH y la legislación laboral.
CRÍTICO Cancelar ausencia borra sustituciones ajenas
Si EVA y SARA están de vacaciones la misma semana y se cancela la de EVA, se eliminan también las sustituciones de SARA para esos días en esa tienda.
IMPORTANTE Sin distinción de contrato GV/IS para compartidos en ficha
El formulario tiene un solo campo "Contrato h/sem", pero internamente hay contratoGV y contratoIS separados. Al editar un compartido, solo se modifica la tienda actual.
IMPORTANTE CAROLINA: restricción de cierre es solo aviso
El sistema avisa pero permite asignarla a cierre en Isabel. Contradice el concepto de "restricción" y puede llevar a errores operativos.
IMPORTANTE 30 días de vacaciones fijos para todos
(Línea 2485) No contempla antigüedad, convenio colectivo, ni contratos a tiempo parcial. Un empleado de 15h/semana no debería tener los mismos 30 días que uno de 40h.

B) Mejoras Técnicas

IMPORTANTE Tipo "asuntos propios" añadido dinámicamente
(Líneas 2726-2734) Se inyecta en el DOM al cargar pero no está en el HTML original. Si el DOM se re-renderiza, se pierde.
IMPORTANTE No hay log de actividad visible
El usuario no puede ver un historial de "qué cambié hoy" sin consultar Google Sheets directamente.

C) Mejoras Funcionales

CRÍTICO Falta gestión de festivos
No hay calendario de festivos. Un lunes festivo aparece como día laborable normal y genera alertas de "bajo mínimos" incorrectas.
CRÍTICO No hay validación de ausencias solapadas
Se puede registrar dos ausencias solapadas para el mismo empleado (vacaciones del 1-15 y baja del 10-20), corrompiendo el conteo de días.
IMPORTANTE Sin resumen mensual de horas por empleado
El tooltip muestra horas semanales al pasar el ratón, pero no hay tabla consolidada mes completo para control de nóminas.
IMPORTANTE No se puede planificar a futuro
Se puede navegar meses, pero no hay modo "planificación" vs "operativo". Falta distinguir entre horarios confirmados y provisionales.
IMPORTANTE No hay flujo para intercambios de turno
Caso frecuente: "SARA y ELI quieren intercambiar el jueves". Actualmente requiere ausencia ficticia + sustitución, cuando debería ser un flujo directo.
IMPORTANTE Falta exportación de datos para nóminas
No hay export CSV/Excel con horas reales trabajadas por empleado y mes, separadas por empresa (KIRA / REYPIK), para alimentar el sistema de nóminas.
IMPORTANTE Sin alerta proactiva de exceso de horas
Solo se ve al pasar el ratón. Debería haber un panel de alertas visible: "EDU lleva 42h esta semana (contrato: 30h)".
DESEABLE No hay gestión de turnos partidos
Si un empleado trabaja 7-10 y 16-20, no hay forma de representarlo. El sistema asume turno continuo.
DESEABLE Falta campo de motivo detallado en ausencias
Para bajas, faltaría "fecha prevista de reincorporación"; para permisos, tipo (paternidad, mudanza, etc.).
DESEABLE No hay control de horas extraordinarias acumuladas
Las horas extra por sustituciones se ven por semana pero no hay acumulado mensual/anual ni alertas legales.

D) Innovaciones

CRÍTICO Notificaciones automáticas
Cuando se registra una ausencia o se asigna un sustituto, debería notificarse al empleado afectado (WhatsApp Business API, email, o app móvil).
IMPORTANTE Predicción de necesidades de cobertura
Basándose en el historial de ausencias, predecir semanas con alta probabilidad de necesitar extras (puentes, Navidad, agosto).
IMPORTANTE Portal de empleado (self-service)
Vista simplificada donde cada empleado vea su horario, solicite vacaciones, y confirme sustituciones.
IMPORTANTE Integración con sistema de fichaje
Comparar horas planificadas vs horas reales fichadas para detectar discrepancias.
DESEABLE Dashboard de KPIs
Tasa de cobertura, horas extra acumuladas, días de vacaciones pendientes por equipo, comparativa entre tiendas.
DESEABLE Exportación directa a software de nóminas
Integración con A3Nom, Sage, o similar.

4. Especialista en UX/UI

A) Errores y Bugs

CRÍTICO prompt() y alert() para interacciones críticas
Se usan diálogos nativos del navegador para editar horarios (línea 2406), dar motivos de descarte (línea 3619), y rechazar propuestas (línea 4041). Sin validación, sin cancelación parcial, y en móvil son especialmente problemáticos.
IMPORTANTE Tooltip de horas solo visible en hover
Los indicadores de exceso/déficit de horas (puntos verde/naranja/rojo) solo muestran detalle al pasar el ratón. En dispositivos táctiles no hay hover y la información es inaccesible.
IMPORTANTE Calendario no es responsive
El layout con col-fds width: 280px y la estructura flex no colapsan en pantallas pequeñas. En tablet o móvil, el calendario se desborda horizontalmente.
IMPORTANTE Animación de alertas agresiva
El parpadeo continuo de los botones "sin cubrir" (animation: centelleo 1s infinite) distrae y genera fatiga visual, especialmente cuando hay muchas alertas simultáneas.

B) Mejoras Técnicas

IMPORTANTE No hay estados de carga en las acciones
Al guardar en Sheets, solo cambia el indicador de sync en el header. No hay feedback en los botones ni spinners en los modales.
IMPORTANTE Modales sin accesibilidad
Los modales no capturan el foco (tab trap), no se cierran con Escape, no tienen aria-* attributes, ni role="dialog".
DESEABLE No hay skeleton loading
Al cargar desde Sheets, la página muestra datos hardcoded y luego "salta" cuando llegan los datos reales.

C) Mejoras Funcionales

CRÍTICO Flujo de sustitución de 5 pasos excesivo
Registrar una ausencia de un día requiere: seleccionar empleado → tipo → fechas → sustituto → confirmar. Para una baja de un día deberían ser 2 clicks, no 5.
IMPORTANTE No hay búsqueda de empleados
Con 15+ empleados por tienda, no hay campo de búsqueda ni filtro en ningún selector.
IMPORTANTE Colores no accesibles para daltonismo
Descarga (azul), mañanas (verde), tardes (naranja), cierre (morado) no se distinguen todos en protanopia/deuteranopia. Falta un patrón secundario (rayas, puntos) o labels siempre visibles.
IMPORTANTE Menú contextual usa click izquierdo
Se abre al hacer click normal en un turno, lo que entra en conflicto con la expectativa de "ver detalle". El usuario no sabe que al clickar un turno aparecerá un menú flotante.
IMPORTANTE Sin confirmación visual al completar acciones
Tras asignar un sustituto o guardar una ausencia, el modal se cierra sin toast/snackbar de confirmación. El usuario no tiene certeza de que se guardó.
DESEABLE Botones del header saturados
6 botones en el header compiten por atención. Falta jerarquía visual clara entre acciones primarias y secundarias.
DESEABLE Formulario de empleado plano
Todos los campos en un solo formulario. Sería más claro separar en tabs: Personal / Laboral / Restricciones.

D) Innovaciones

IMPORTANTE Drag & drop para reasignar turnos
Arrastrar un empleado de una franja a otra o de un día a otro.
IMPORTANTE Vista timeline (tipo Gantt) por día
Visualizar las horas de cada empleado como barras horizontales en una línea de tiempo, para ver solapamientos y huecos de un vistazo.
IMPORTANTE Tema oscuro
Para uso prolongado, especialmente si se consulta en turno de noche (cierre).
DESEABLE Atajos de teclado
N para nueva ausencia, ←/→ para cambiar mes, 1/2 para cambiar tienda, Esc para cerrar modales.
DESEABLE Onboarding / ayuda contextual
Primera vez que se abre, un tour guiado que explique las funciones principales.
